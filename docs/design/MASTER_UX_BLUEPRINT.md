## 들어가며 — 이 문서의 위치

본 문서는 꽃틔움 가든의 **두 번째 골격**이다. 첫 번째 골격(Phase A~E + Workflow A1a~A3-4a)이 "기능을 갖춘 앱"을 만들었다면, 이 블루프린트는 "1인 셀러의 하루를 끝내주는 운영 시스템"을 정의한다. 시각 디자인(색상, 타이포, 컴포넌트 디테일)은 별도 문서에서 다룬다.

**핵심 질문**: *1인 운영자가 매일 아침 이 앱을 열었을 때, 가장 짧은 동선으로 가장 많은 매출을 만들도록 하려면 어떤 화면이 필요한가?*

## Part 1. 진단 (Audit) — 1인 셀러의 5대 페인포인트

B2B ERP/백오피스를 설계할 때는 보통 *역할 분리*가 전제된다 — MD가 등록하고, CS가 응답하고, 물류가 발송한다. 그러나 **1인 셀러는 이 모든 역할을 혼자**하며, 그것도 본업(디자인, 제작, 재택 등) 사이의 자투리 시간에 처리한다. 이 구조적 차이가 5대 페인포인트를 만든다.

### P1. 컨텍스트 스위칭 비용
하루에 평균 **8~12회** 다른 작업으로 점프한다 — 아침에 주문 확인 → 점심 후 신상품 등록 → 저녁에 CS 응답 → 자기 전 트렌드 체크. 매번 "지금 뭘 해야 하지?"를 인지하는 데 평균 **30~90초**가 든다. 이 시간이 누적되면 하루 **10~15분**의 손실이다.

> 설계 함의: 화면을 열면 **3초 안에 "지금 해야 할 가장 중요한 1가지"가 보여야** 한다.

### P2. 누락 위험 (silent failure)
파워셀러 3대 KPI(구매확정률 / 리뷰수 / 굿서비스)는 모두 **누적 시간 윈도우**로 측정된다. D+3 안에 알림을 못 보내면 구매확정률이 떨어지고, D+30 안에 한달리뷰를 유도 못 하면 리뷰수가 멈춘다. 이런 누락은 *경고가 안 뜬다* — 그저 KPI가 천천히 떨어진다.

> 설계 함의: 모든 시간 윈도우를 **타임축 위젯**으로 가시화하고, 윈도우 진입 N건이 0보다 클 때만 노출. 0건이면 위젯을 숨기지 말고 "0건, 잘하고 계세요" 같은 안심 상태로 보여줘 누락 여부를 매일 확인 가능하게 한다.

### P3. 등록 품질 = 매출 직결
스마트스토어 검색 알고리즘은 **등록 품질**(키워드, 카테고리, 이미지, 마진, SEO 태그)을 노출 확률에 직접 반영한다. 1인 셀러는 등록 시점에 모든 항목을 100% 채울 시간이 없다. **"60%로 빨리 올리고 나중에 다듬자"가 만든 좀비**가 정원 창고에 쌓인다.

> 설계 함의: 등록 준비도(11항목 점수)를 **단순 표시가 아닌 액션 가능한 형태**로. 각 항목의 부족분을 "AI가 채울 수 있는 것 / 셀러만 가능한 것"으로 분리해 보여준다.

### P4. 데이터의 비대칭 (Source of Truth 분산)
같은 정보가 **여러 위치**에 흩어진다 — 카카오 채널 ID는 store_settings에, 솔라피 키도 store_settings에, 배송 템플릿은 별도 테이블에, 공급사 정보도 별도. 셀러가 한 번에 "내 가게 설정 전체"를 볼 곳이 없다.

> 설계 함의: **단일 설정 허브**(/settings/store)에서 모든 마스터 데이터를 한 화면으로 조회/편집. 카카오/네이버/배송/공급사는 그 허브의 자식 화면으로 이동.

### P5. 자동화의 가시성 부족
cron이 매일 08:00 KST에 7가지 작업(OOS / score / sourcing / autoConfirm / competition / Naver sync / 추천)을 돌린다. 그러나 셀러는 **언제 어떤 결과로 돌았는지** 알 수 없다. Discord에 알림이 가지만, 알림은 *지금 일어난 일*이지 *시스템이 잘 돌아가고 있다는 증거*가 아니다.

> 설계 함의: 대시보드 어딘가에 **자동화 헬스 카드** — 마지막 cron 실행 시각, 7가지 작업의 성공/실패, 다음 실행 예정 시각.

## Part 2. 설계 원칙 7개 (Design Principles)

이 7개는 본 블루프린트의 모든 화면에 일관되게 적용된다. 디자인을 손볼 때 이 원칙이 흔들리면 안 된다.

### DP1. 시간축 우선 (Time-axis First)
시간 윈도우(D+3~5, D+8, D+28~32) 기반 작업은 **시간축 위젯**으로 표현. 카운트만 보여주지 말고 "D+3 (긴급), D+4 (보통), D+5 (마지막 기회)" 식으로 윈도우 내 시급도까지 시각화. 이미 적용 중인 패턴: `ConfirmationReminderWidget` D+3·4·5 컬러 분기(yellow→orange→red).

### DP2. 자동화 우선, 수동 마감 (Automate First, Finalize Manually)
**원칙**: 시스템이 가능한 모든 자동 처리를 먼저 하고, 셀러는 *결정만* 한다. 등록 = AI 자동 채우기 7개 항목 → 셀러는 미리보기 모달에서 체크박스 7개로 적용/거부. 발주확인 = cron이 4시간마다 자동 처리 → 셀러는 예외만 본다.

### DP3. 단일 화면 단일 결정 (One Screen, One Decision)
한 화면에 여러 결정이 필요하면 분리. 대시보드 = 오늘 무엇을 할지. 정원 창고 = 어떤 상품을 어떻게 다룰지. 검색 조련사 = SEO를 어떻게 강화할지. 화면 사이의 점프는 deep-link로 이어가되, *정보 단위 자체는 분리* 유지.

### DP4. 컨텍스트 보존 (Deep-link First)
모든 위젯의 액션은 **컨텍스트를 가진 deep-link**로 연결. 예: 등록 준비도 위젯의 "이미지 부족" 칩 → `/products/new?focus=image&productId=X` (씨앗심기 이미지 탭 자동 활성화 + 해당 상품 자동 선택). 이미 적용 중: E-14 ITEM_TO_TAB 11개 매핑.

### DP5. 파이프라인 가시화 (Pipeline Visibility)
1인 셀러의 머릿속에는 *모든 단계가 동시*에 떠 있다 — 소싱 12개, 등록 대기 8개, 판매 중 6개, 좀비 3개. 사이드바 배지로 **각 단계의 카운트**를 항상 노출. 클릭 한 번에 해당 단계로 이동.

### DP6. 매출 임팩트 정렬 (Revenue-Impact Sorted)
모든 리스트는 *매출에 가까운 순*으로 정렬. 등록 준비도는 **점수 낮은 순**(작업 필요한 것 먼저), 정원 창고는 **라이프사이클 단계 순**(PEAK → DECLINING → ZOMBIE), 좀비 부활소는 **부활 잠재력 순**(키워드 트렌드 상승 중인 것 먼저).

### DP7. Empty State는 안심 메시지 (Empty State as Reassurance)
0건은 **위험 신호일 수도, 잘 하고 있다는 신호일 수도** 있다. "오늘 발송할 주문 0건" → "아직 주문이 없어요, 트렌드 키워드를 확인해보세요." "D+3 알림 대상 0건" → "어제 모두 발송 완료되었어요." Empty state도 *다음 행동 제안*과 함께.

## Part 3. 정보 아키텍처 (IA)

### 3.1 현재 IA 진단 (꽃틔움 가든 v1.0)

```
GARDEN  → 정원 일지 (대시보드)
HUNT    → 꿀통 사냥터 (소싱)
PLANT   → 씨앗 심기 (상품 등록)
TEND    → 정원 창고 / 검색 조련사 / 좀비 부활소
ORDERS  → 주문 관리
OPS     → 인서트 카드
TOOLS   → 거래처 명단 / 배송 레시피 / 공급사 열쇠방 / 카카오 채널 / 네이버 기본값
```

**강점 (유지할 것)**:
- 메타포 일관성 — 가든 컨셉(정원/씨앗/좀비/꼬띠)이 모든 메뉴에 일관되게 적용. 시각 디자인 시 강력한 자산
- 기능 그룹핑 — GARDEN/HUNT/PLANT/TEND/ORDERS/OPS/TOOLS 7섹션은 1인 셀러의 작업 흐름과 정확히 일치
- 사이드바 배지 — sourcing/draft/zombie/orders 4가지 배지로 P5(파이프라인 가시화) 일부 충족

**약점 (개선 후보)**:
- OPS 섹션이 너무 가볍다 — 인서트 카드 1개만 있어 섹션의 존재 의미가 약함. 나중에 운영 일지/일일 마감 등 OPS 도구가 추가되면 자연스러워질 것이지만, 지금은 흡수 검토 가능
- TOOLS의 5개 메뉴가 평면 나열 — 거래처/배송/공급사 열쇠방은 *마스터 데이터*, 카카오/네이버는 *외부 채널 연동*으로 성격이 다름. 그룹화 필요
- /products/sourced와 /sourced 중복, /products/upload, /upload 중복 — 라우트 정리 필요 (어느 한쪽으로 일원화)
- /workflow, /portfolio, /search-products — 사이드바에 없으나 페이지가 존재. 사용 빈도 검토 후 사이드바 진입 또는 deprecation 결정

### 3.2 권장 IA v2 (작업원칙 #27 — 기존 메뉴 0 삭제, 그룹화만 조정)

```
GARDEN — 정원 일지 (Dashboard)
   /dashboard

HUNT — 매출 후보 발굴
   /crawl              꿀통 사냥터 (소싱 검색)

PLANT — 등록 워크플로우
   /products/new       씨앗 심기 (개별 등록)
   /products/upload    대량 등록 (현 /upload 일원화)

TEND — 라이프사이클 관리
   /products           정원 창고 (전체 상품)

ORDERS — 주문/CS

SETTINGS — 가게 설정 (TOOLS 재구조화)
   /settings/store              가게 정보 (신규 — 단일 설정 허브)
   Master Data 그룹
       /settings/suppliers      거래처 명단
       /settings/shipping       배송 레시피
       /settings/supplier-login 공급사 열쇠방

       /settings/kakao          카카오 채널
       /naver-settings          네이버 기본값
```

**v1 대비 변화**:
1. OPS → ORDERS 흡수: 인서트 카드는 주문 발송 시 동봉되는 자산이므로 ORDERS와 한 묶음이 자연스럽다
2. TOOLS → SETTINGS로 의미 명확화: "도구"는 모호한 라벨, "설정"은 의도가 명확
3. SETTINGS 안에 Master Data / External Channels 2그룹: 거래처/배송/공급사 열쇠방은 *내가 가진 데이터*, 카카오/네이버는 *외부 시스템 연결*. 시각 분리
4. /settings/store 신규 추가: 가게 정보 단일 허브 (P4 페인포인트 해결)
5. /sourced, /products/upload 라우트 일원화: 중복 페이지 정리

### 3.3 메뉴 라벨링 정책 — 메타포 vs 직설

**결론**: **현 메타포(가든 컨셉) 유지 권장**.

근거:
- 1인 셀러는 *친근감*이 동기 부여. "씨앗 심기"는 "신상품 등록"보다 심리적 진입 장벽이 낮다
- 메타포가 일관되면 학습 비용은 한 번에 끝난다 (꽃졔님은 이미 학습 완료)
- 단, **첫 사용자**에게는 메타포가 장벽일 수 있으므로 사이드바 hover/focus 시 부제목으로 직설 라벨 노출 — 예: "씨앗 심기 (신상품 등록)"

## Part 4. 핵심 화면 8개 설계

각 화면은 다음 5요소로 정의한다:
- **Job to be done (JTBD)**: 셀러가 이 화면에서 정확히 무엇을 끝내려 하는가
- **정보 계층**: 위 → 아래 우선순위
- **주요 액션 (CTA)**: 화면당 1~3개
- **자동화 트리거**: 시스템이 미리 처리하는 것
- **1인 셀러 효율 패턴**: 시간 절약 또는 누락 방지

### 화면 1. 정원 일지 (Dashboard) — 일일 명령탑

**JTBD**: 아침에 앱을 열고 *오늘 처리할 작업의 우선순위*를 5초 안에 파악한다.

**정보 계층 (today 모드 기준)**:
```
+------------------------------------------------------+
| Section 1: KPI Strip (4)                             |
|  오늘 주문 N / 매출 W / 진행 N / 굿서비스 등급      |
+------------------------------------------------------+
| Section 2: ACTION (today 모드 only)                  |
|  - 자동화 헬스 카드 (cron 마지막 실행, 7가지 결과)   | <- 신규 제안
|  - 등록 준비 명령탑 (E-14, DRAFT TOP 5 + AI 채우기)  |
|  - 구매확정 리마인더 (D+3~5, A3-1b)                  |
|  - 한달리뷰 위젯 (D+28~32, A3-4b 예정)               | <- B-1 작업
|  - 이벤트 타임라인 (최근 24h, A3-2)                  |
+------------------------------------------------------+
| Section 3: MARKET (트렌드/경쟁, mode별 정렬 변경)    |
|  꼬띠 브리핑 / 마켓 트렌드 / DataLab / 경쟁 / 소싱   |
|  라이프사이클 / 굿서비스 / 수익성                    |
+------------------------------------------------------+
| Section 4: TOOLS (빠른 작업)                         |
|  씨앗심기 / 검색조련사 / 주문관리 / 꿀통사냥터       |
+------------------------------------------------------+
```

**주요 액션 (today 모드)**:
1. 등록 준비 명령탑 → "AI 채우기" 또는 "바로 등록"
2. 구매확정/한달리뷰 위젯 → 발송 또는 카카오 채널 설정
3. 자동화 헬스 → 비정상 시 진단 페이지로 이동

**자동화 트리거**:
- daily cron 08:00 KST → OOS / score / sourcing / autoConfirm / competition / Naver sync / 추천 7가지
- weekly cron → 주간 수익 보고서 Discord 발송
- 5분 SWR → 5개 위젯 (GoodService / ReviewGrowth / Competition / ConfirmationPending / EventTimeline / MonthReview)
- 60초 SWR → 7개 위젯 (Sidebar / Profitability / ProductsList / UploadReadiness / ProductLifecycle / DashboardStats / NaverSeoProducts)
- 24h SWR → 2개 (DataLabTrend / SourcingRecommend)

## Part 5. 핵심 사용자 여정 4가지

### 여정 1. 신규 상품 등록 (소싱 → 등록 → 공개)

**시간 예산**: 1인 셀러 기준 신상품 1개당 **30분 → 5분** 단축이 목표.

```
[Day 1 아침]
1. 정원 일지 → 꼬띠 추천 위젯에서 "트렌드 키워드 5개" 확인
2. 키워드 클릭 → 꿀통 사냥터로 이동 (deep-link, 키워드 prefilled)
3. 도매꾹 검색 결과 카드 → 진입장벽 LOW 칩 + 마진 30%+ 후보 3개를 보관함에 담기
        → 5분

[Day 1 저녁]
4. 보관함 → "씨앗 심기로 이동" → 후보 1개씩 등록
5. 씨앗 심기 → AI 자동 채우기 (7항목 한 번에) → 미리보기 모달 → 7개 체크박스로 적용
6. 부족한 4항목(이미지 2장 + 배송 템플릿 + 마진) 직접 입력
7. 등록 준비도 90+점 도달 → "이대로 등록" CTA → 네이버 직접 등록 또는 엑셀 다운로드
        → 후보 1개당 5분

[Day 2 아침]
8. 정원 일지 → 등록 준비 명령탑에서 어제 등록한 상품 확인 (cron이 점수 재계산)
9. 검색 조련사 이동 → 꼬띠 AI 3버튼 중 "정석SEO" 적용 → 전체 저장
        → 1분
```

**총 시간**: 후보 3개 등록 시 약 **20분** (기존 90분 대비 78% 단축).

[18:00] 마감
9. 정원 일지 week 모드 → 이번 주 매출 확인

### 여정 3. 리뷰/매출 성장 (D+0 → D+3~5 → D+28~32)

```
[D+0] 결제 완료
- 시스템: orders 테이블에 신규 주문 row 생성, status=PAID

[D+1~D+3] 발송
- 셀러: 송장 등록 (B-2)
- 시스템: status=SHIPPING

[D+3~D+5] 구매확정 유도 (E-2B 초록 뱃지)
- 셀러: orders 테이블의 초록 뱃지 클릭 → 알림톡 또는 네이버 내장 알림
- 자동: 네이버 기본 무료 리뷰 알림 (배송완료 3일 후 자동 발송)

[D+8] 자동 구매확정 (네이버 기본)
- 시스템: status=CONFIRMED

[D+9~D+11] 리뷰 요청 (E-2B 파랑 뱃지)
- 셀러: 파랑 뱃지 클릭 → 알림톡
- 자동: 네이버 구매확정 시 리뷰 작성 알림 (자동 발송)

[D+28~D+32] 한달리뷰 유도 (E-2B 보라 뱃지) <- B-1 작업 영역
- 셀러: 정원 일지 한달리뷰 위젯 + orders 보라 뱃지에서 일괄 알림톡
- 효과: 리뷰 볼륨 1.5~2배 (초기 + 한달리뷰)
```

**개선 제안**:
- 리뷰 작성률 KPI — 정원 일지에 "이번 달 리뷰 작성률 N%" 표시. 목표 20~25%
- 재구매 유도 D+45~60 — 한달리뷰 이후 재구매 알림 윈도우 (장기 추가 검토)

### 여정 4. 좀비 상품 부활 (DECLINING → ZOMBIE → reactivation)

```
[자동 분류]
1. E-3 cron이 매일 라이프사이클 갱신
2. 30일 무판매 → DECLINING, 60일 → ZOMBIE
3. Discord 알림 (D-3 패턴 재사용 가능)

[셀러 결정]
4. 좀비 부활소 → 부활 잠재력 정렬
5. 트렌드 상승 중 + 진입장벽 LOW로 변화한 상품 → "Clone for 재등록"

[Clone 워크플로우]
6. 신규 상품 ID로 복제 (기존 ID는 SOLDOUT)
7. 씨앗 심기 자동 이동 + 기존 데이터 prefilled
8. SEO 재작성 (꼬띠 AI 3버튼)
9. 이미지 교체 (원래 약점이었던 항목)
10. 90+ 도달 시 등록

[효과]
- 네이버 검색 알고리즘이 *새 상품으로 인식* → 노출 회복
- 기존 상품의 부정 시그널(긴 무판매 기간) 끊고 새로 시작
```

## 부록 A. 컴포넌트 라이브러리 권장 항목

향후 시각 디자인 시 다음 컴포넌트를 표준화한다:

### 데이터 표시
- `<KpiCard>` — 4종 (revenue, count, ratio, grade)
- `<WidgetCard>` — 모든 대시보드 위젯의 컨테이너
- `<EmptyState>` — title / description / illustration / CTA
- `<Chip>` — 5 variants (status, lifecycle, time-axis, supplier, priority)
- `<Badge>` — 사이드바/탭 카운트
- `<HealthIndicator>` — green/yellow/red 점

### 입력
- `<TabBar>` — 6개 탭 패턴 (씨앗 심기) / 5개 탭 (정원 창고)
- `<FilterTabs>` — 카운트 포함 필터
- `<MultiSelectTable>` — 다중 선택 + 일괄 액션
- `<InlineEditor>` — 셀 더블클릭 편집

### 액션
- `<ActionBar>` — 다중 선택 시 노출되는 일괄 액션 바
- `<DeepLinkButton>` — 화면 점프 + 컨텍스트 전달
- `<AICTAButton>` — Sparkles 아이콘 + 보라 톤 (AI 액션 표준)
- `<RefreshAction>` — RefreshCw 아이콘 + isValidating 시 회전

### 피드백
- `<Toast>` — success/error/info/warning
- `<ConfirmModal>` — 위험 액션 확인 (삭제/일괄 변경)
- `<PreviewModal>` — AI 채우기 미리보기 표준

## 부록 C. 모바일 고려사항 (Tier 3 — 향후)

1인 셀러는 외출 중에도 *주문 확인*과 *알림 응답*이 필요. PWA로 다음 화면만 모바일 최적화:

1. 모바일 대시보드 — 오늘 주문 N + 시간축 알림 N + 자동화 헬스
2. 모바일 주문 관리 — 주문 카드 (테이블 X) + 슬라이드 액션
3. 모바일 알림 센터 — Discord/카카오/Toast 통합 피드
4. 모바일 카메라 연동 — 송장 OCR / 도매꾹 상품번호 QR 스캔 (Tier 3)

데스크톱 우선 유지: 등록/SEO 작업은 모바일에서 비현실적.

## 마치며

본 블루프린트는 *현재 꽃틔움 가든의 강점을 보존하면서* 1인 셀러 ERP/백오피스의 표준 패턴(파이프라인 가시화, 시간축 위젯, 자동화 헬스, 단일 허브)을 결합한 v1 설계서다.

**핵심 인사이트 3가지**:
1. 꽃틔움 가든의 가든 메타포는 그대로 유지하되, 정보 구조는 더 명확히 그룹화한다 (TOOLS → SETTINGS의 2단계 그룹)
2. 자동화는 이미 충분하나 가시성이 부족하다 — 자동화 헬스 카드 + 운영 일지 + 알림 센터로 셀러의 신뢰를 만든다
3. 시간축 기반 위젯이 1인 셀러 운영의 핵심 — 구매확정/한달리뷰/좀비 진입 등 모든 주요 KPI는 시간 윈도우로 측정되므로 시간축 시각화가 모든 화면의 표준이 되어야 한다

**다음 단계 (꽃졔님 결정)**:
- A. 본 블루프린트 v1 검토 + 피드백 → v2 진화 (디자인 손보기 전 IA/UX 확정)
- B. Tier 1 디자인 토큰 작업 시작 (Design Tokens + 컴포넌트 라이브러리)
- C. 즉시 적용 가능 UX 개선 10선 중 우선순위 1~3개 구현 (자동화 헬스 / 단일 설정 허브 / Cmd+K)
- D. 보류 중이던 B-1 (MonthReviewWidget) 진행 (본 블루프린트의 Section 2 today에 자연스럽게 자리 잡음)

본 문서는 *살아있는 문서*로 운영한다. 디자인/구현 진행 중 학습한 패턴은 이 블루프린트로 다시 흘러들어와 v2, v3로 진화한다.
