# ORDER-QUEUE — 처리 필요 주문 → 대시보드 액션 서피스 통합 스펙 (2026-07-06)

Authoring: DESKTOP. 구현=Code. 권위: 본 문서 + OPERATOR_SYSTEM_BLUEPRINT(#56) + 대시보드 액션 서피스 + Order 테이블(ORDER-SYNC로 정확 동기화됨). 원칙 #55/#56/#181/#46.
전제: ORDER-SYNC(3-endpoint·prod검증) DONE → Order.status 정확. 이제 처리 필요 주문을 운영 흐름에 surface.

================================================================
## 0. 목적 (왜)
================================================================
ORDER-SYNC로 주문 상태가 정확해졌으니, "셀러가 지금 처리해야 할 주문"(발송 대기·클레임 응대)을 대시보드 액션 서피스에 능동 표면화 → 1인 셀러가 주문 페이지를 뒤지지 않고 큐만 보고 즉시 처리. 주문 관리 루프 완결(입고 동기화 → 처리 넛지 → 액션).

================================================================
## 1. 설계 판단 (통합 지점)
================================================================
- C-9 Operator Action Queue(control-tower-engine computeActionQueueItem)는 상품별(Product) 매트릭스 → 주문(Order)은 별개 엔티티라 부적합.
- 정합 지점 = 대시보드 "오늘 처리해야 할 액션" 서피스(현재 DRAFT 등록/품절 보충/발주 처리 표시, 주문 미포함). 여기에 주문 처리 항목 추가.
- verify-first(#181): Code가 대시보드 액션 서피스 컴포넌트/데이터 소스(어느 라우트가 "오늘 처리 액션"을 계산하는지) 실측 후 주문 항목 배선.

================================================================
## 2. 트리거 (Order.status 기준·읽기 전용)
================================================================
| status | 의미 | action |
|---|---|---|
| PAID | 결제완료·미발송 | 발송 필요 (최우선) |
| CANCEL_REQUESTED | 취소 요청 | 응대 필요 |
| RETURN_REQUESTED | 반품 요청 | 응대 필요 |
| (opt) DELIVERED N일 경과 | 구매확정 지연 | 확정 유도(자동이면 생략) |
- 현 실측: PAID 1건(발송 대기). 요청형 클레임은 발생 시 표면화.
- 판정=읽기 전용(Order 조회). 신규 컬럼 0.

================================================================
## 3. UI / 액션
================================================================
- 대시보드 액션 서피스에 넛지: "발송 필요 N건" / "클레임 응대 N건"(0이면 미표시).
- 1클릭 딥링크 → /orders(해당 상태 필터) → 기존 dispatch/confirm/claim 액션(이미 구현·별도 라우트).
- 비강제·건수 배지. 한글 i18n 격리(#3-1)·이모지0·Lucide.

================================================================
## 4. 단계
================================================================
- ORDER-QUEUE-1: 대시보드 액션 서피스에 발송/클레임 넛지(건수+딥링크). 읽기 판정. → 검증: PAID 1건→"발송 필요 1건" 노출·딥링크·0건 시 미표시.
- ORDER-QUEUE-2(후속): 넛지에서 인라인 발송 처리(dispatch 원클릭) — GO-게이트(#46 쓰기).

================================================================
## 5. 원칙·규격
================================================================
- 발송/확정/클레임 응대 = 쓰기(#46) — 액션 실행은 운영자 클릭. 넛지 판정은 읽기.
- 주문 처리 개입은 대시보드 서피스(상품 C-9 매트릭스 아님) — 엔티티 구분(#195).
- 전주문 범용·기존 dispatch/confirm/claim 라우트 재사용(신규 배관 0·#181).

================================================================
## 작업 유의사항/원칙
================================================================
- #195 주문(Order) 개입은 상품(Product) C-9 매트릭스가 아니라 대시보드 액션 서피스에 배치 — 엔티티가 다르면 개입 서피스도 분리(#56 개입점을 엔티티에 맞게).
- #181: 대시보드 "오늘 처리 액션" 계산 소스를 실측 후 주문 항목 확장(신규 서피스 신설 금지).
- 기존 주문 액션 라우트(dispatch/confirm/claim) 재사용 — 넛지는 진입점만 제공.
