# 발행 준비 상태(Publish Readiness) 앱 surface — 스펙 (2026-07-10)

Authoring: DESKTOP(코드 실측 기반 설계). 계기: 운영자 원칙 "발행 지장 정보를 앱에 심어 독자 사용 시 문제없게(개입점을 앱에 녹임)". 권위: product-builder.ts validateForRegistration. 원칙 #46/#55/#240. 상태: 스펙 완료·Code 구현 대기(독립).

## 0. 문제 (현행)
- `validateForRegistration`(product-builder.ts:806)이 발행 시 8가지를 HARD BLOCK로 검증하지만, **발행 시도 시 에러 메시지로만** 노출 → 운영자가 사전에 "뭐가 준비됐고 뭐가 막혔는지" 앱에서 못 봄.
- 결과: 운영자가 발행 가능 여부를 스스로 판단 못 하고 외부(나) 분석에 의존. → 앱 독자 사용성 저하.

## 1. 검증 항목 (실측·8종 HARD BLOCK)
| # | 체크 | 실패 조건 |
|---|---|---|
| 1 | 상품명 | 5자 미만 |
| 2 | 카테고리 | 유효 8자리 리프 코드 아님 |
| 3 | 대표이미지 | 없음 |
| 4 | 판매가 | 0 이하 |
| 5 | 주소 동기화 | 네이버 주소 ID 미동기화 |
| 6 | 업로드 준비도 | readiness score 임계 미만 |
| 7 | 단위가격 | 대상 카테고리인데 필드 없음(비대상이면 통과) |
| 8 | 원산지 | 미설정/추정(#95 법적 리스크) |

## 2. 설계 — 앱 개입점 (product-agnostic·systemic)
**A. 로직 계층**: `getPublishReadiness(product)` 신설 — validateForRegistration의 검증을 재사용하되 **구조화 결과** 반환(에러 문자열 아님):
```
{ ready: boolean, passed: number, total: number,
  checks: [{ key, label, status: 'pass'|'fail'|'na', message, fixHref? }] }
```
- 각 체크에 "고치러 가기" 링크(fixHref: 카테고리→/products/[id] 편집·원산지→해당 섹션·주소→설정 등).
- 단위가격은 비대상이면 status='na'(회색·통과 취급).

**B. UI surface (3곳·중복 아닌 계층)**:
1. **상품 상세/발행 영역**: "발행 준비 X/8" 배지 + 펼치면 체크리스트(pass 초록·fail 코랄·na 회색)+고치기 링크. 발행 버튼은 ready=false면 비활성+사유 표시(#46 GO 게이트 유지).
2. **대시보드 관제탑 요약카드**: "발행 대기 N건·준비완료 M건" 집계(정원 컨셉=망루). 클릭→상품 목록 필터.
3. **오늘 할 일(선택)**: 준비 거의 다 된 상품(1~2개 남음)을 "곧 발행 가능—OO만 채우면 돼요"로 넛지.

**C. 정직 표기**: na(비대상)은 "해당 없음"으로·발행 안전 항목은 초록. 운영자가 앱만 보고 "명화는 단위가격 해당없음·원산지 중국산 설정됨·발행 가능"을 스스로 확인.

## 3. 실데이터 정합 (독자 사용)
- 명화: 단위가격 na·원산지 중국산 설정·카테고리 유효 → 준비 상태 앱에 반영 시 "발행 가능(운영자 GO 대기)"로 표시.
- 3상품 카테고리 전부 유효(5021 table)·단위가격 전부 비대상 → 앱이 이를 자동 판정해 표시.
