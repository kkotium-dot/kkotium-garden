# 상품 관리 통합 허브 — 연동+앱생성 단일 관리 (2026-07-10)

Authoring: DESKTOP. 계기: 운영자 명시 — "상품 연동이 연동 상품 개선/리셋/튜닝/재고/마진 관리만이 아니라, 앱에서 지금부터 올리는(씨앗심기) 상품도 반영돼 하나의 상품 관리 기능이 됐으면". 원칙 #55/#56/#232/#245. 상태: 스펙·Code 구현 대기(독립·부활후보 스펙 상위 통합).

## 0. 운영자 의도 (확정)
- 상품 = 두 출처: (a) **연동 import**(스토어에서 부활 대상 가져옴) (b) **앱 생성**(씨앗심기로 신규 등록). 
- 둘을 **분리된 기능이 아니라 하나의 상품 관리 허브**로 — 개선·리셋·튜닝·재고·마진·발행을 출처 무관 동일하게.

## 1. 현행 (실측)
- 연동 import·앱 생성 **둘 다 로컬 Product 행**으로 저장됨 → DB Product = 이미 단일 SoR.
- 그러나 UI가 분리: /products/link(연동 전용)·/products(목록) 등 → 출처별로 흩어져 "하나의 관리"가 안 됨.
- ★갭: 출처 태그·통합 목록·공통 관리 액션·앱생성 상품 자동 편입이 없음.

## 2. 설계 — 단일 상품 관리 허브 (systemic)
**A. 데이터**: Product에 `origin_kind` 태그(IMPORTED / APP_CREATED / HYBRID). 연동=IMPORTED, 씨앗심기=APP_CREATED. 재업로드된 연동상품이 앱개선 거치면 HYBRID.
**B. 단일 목록 `/products`(허브)**: 출처 무관 전 상품. 각 행에 통합 상태축:
| 축 | 표시 | 소스 |
|---|---|---|
| 출처 | 연동/앱생성/혼합 배지 | origin_kind |
| 등록상태 | 미등록/판매중/판매중지 | #240 강화 |
| 발행 준비 | X/8 | #240 |
| 부활 필요도 | S/A/B/C | #244 revival-score |
| 재고 | 수량/품절 | stock |
| 마진 | %/원 | 마진 계산 |
| drift | 동기화 필요 | drift-scan |
- 필터/정렬: 출처·부활후보·발행대기·품절·마진낮음.
**C. 공통 관리 액션(출처 무관)**: 튜닝(아틀리에)·리셋·재고 조정·마진 재산정·발행 준비·재업로드·부활소 이동.
**D. 앱생성 자동 편입**: 씨앗심기로 만든 상품이 생성 즉시 허브에 APP_CREATED로 등장(별도 등록 불필요).
**E. 흐름 수렴**: (연동 부활후보 선별 import #244) + (앱 신규 씨앗심기) → **동일 허브** → 튜닝/발행 준비/재고·마진 → 발행/재업로드. 두 진입점, 하나의 파이프라인.

## 3. 정원 세계관 매핑
- 허브 = "정원 전체 화단 대장(모든 꽃 한눈에)". 연동=옮겨심은 모종·앱생성=씨앗부터 키운 꽃·부활대상=좀비 부활소 화분. 관제탑/온실과 연계.

## 4. 구현 절차 (Code·독립·단계)
1. Product.origin_kind 마이그레이션(기존 행 backfill: naverProductId 있으면 IMPORTED·없으면 APP_CREATED). nullable 아님 주의.
2. /products 허브 목록에 통합 상태축 컬럼+필터/정렬(부활#244·등록상태#240 재사용).
3. 공통 관리 액션 바(튜닝/리셋/재고/마진/발행 준비/부활소).
4. 씨앗심기 완료→허브 자동 편입 확인. link/import는 허브의 "연동 진입점"으로 재배치(별도 기능 아님).
  tsc0/build0→Desktop 검증(연동·앱생성 상품이 한 목록에·공통 액션 동작·앱생성 자동 편입).

## 작업 유의사항/원칙 (신규)
- **#245** 상품 관리는 출처(연동 import vs 앱 생성)로 분리하지 말고 **단일 허브(DB Product=SoR)**로 통합 — origin_kind 태그로 구분만 하고 목록·상태축·관리 액션은 전상품 공통. 앱에서 새로 만든 상품도 생성 즉시 허브에 편입돼 동일하게 관리(리셋/튜닝/재고/마진/발행). 두 진입점(선별 import·씨앗심기)이 하나의 파이프라인으로 수렴. 기능을 출처별로 쪼개면 운영자 인지부하↑(#232/#56) → 항상 통합 surface.
