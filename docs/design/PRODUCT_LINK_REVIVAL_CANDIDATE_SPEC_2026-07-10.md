# 상품 연동 — 부활 후보(Revival Candidate) 선별 강화 스펙 (2026-07-10)

Authoring: DESKTOP(코드 실측 기반). 계기: 운영자 상품연동 의도 명시 — "전체 import는 무거움 → 좀비부활/리셋튜닝 필요한 상품만 선별 import해 앱으로 개선 후 재업로드". 원칙 #55/#56/#240/#244. 상태: 스펙·Code 구현 대기(독립).

## 0. 운영자 의도 (확정)
- 상품 연동 = 스토어의 모든 상품을 앱에 가져오는 게 아님. **개선(부활)이 필요한 상품만 선별 import** → 앱(좀비 부활소·아틀리에)에서 리셋/튜닝 → 스토어에 재업로드.
- 목적: 죽어있는/부진한 상품을 되살려 매출 회복.

## 1. 현행 구조 (실측)
- import route(PL-1): **선택 상품만 import**(id 목록·빈 목록 거부). statusType→app status 매핑(SALE·OUTOFSTOCK·SUSPENSION→INACTIVE).
- link page: Zone1 스토어 상품 목록(선택) + Zone2 연동 목록(sync/drift 배지·imported/native/conflict 필터) + drift-scan(앱↔네이버 불일치 감지).
- ★갭: 선별 import는 되지만, **"어떤 상품이 부활 대상인지" 자동 식별이 없음** → 운영자가 눈으로 골라야 함.

## 2. 강화 — 부활 후보 스코어링 (systemic·product-agnostic)
스토어 상품 목록에 **부활 필요도 점수 + 배지 + 정렬/필터** 추가. 가용 신호(네이버 API 제공)로 스코어:
| 신호 | 부활 필요 방향 | 데이터 소스 | 가중 |
|---|---|---|---|
| statusType=SUSPENSION | 판매중지=죽은 상품·최우선 부활 | 상품 목록 API | 높음 |
| statusType=OUTOFSTOCK | 품절=기회손실 | 상품 목록 API | 높음 |
| 재고 0/저재고 | 재입고·대체 필요 | 상품 목록 API | 중 |
| 상품명 SEO 품질 낮음(짧음·키워드 부족·특수문자) | 리네이밍 튜닝 대상 | 상품명(로컬 계산) | 중 |
| 대표이미지 1장/저품질 | 이미지 개선 대상 | 이미지 수 | 중 |
| 가격 경쟁력(카테고리 평균 대비 이탈) | 가격 튜닝 | 카테고리 시세(제한적) | 하 |
| 등록 후 경과·최근 수정 없음 | 오래 방치=리프레시 | 등록/수정일(가용 시) | 하 |
- ★리뷰수는 네이버 리뷰 조회 API 미제공(#231) → 스코어에서 제외(우회 불가·크롤링 금지).
- 점수→등급(부활시급 S/A/B/C). 목록에 배지+"부활 후보" 필터+시급순 정렬.

## 3. 워크플로우 통합 (#56 개입점)
- 부활 후보 목록 → 선택 import → **자동으로 "좀비 부활소" 큐에 적재** → 아틀리에 튜닝(SEO 리네이밍·이미지 개선·가격 재산정) → 발행 준비 체크(#240) → 재업로드(운영자 GO).
- 관제탑/대시보드에 "부활 후보 N건" 넛지(정원 컨셉=좀비 부활소).
- drift-scan과 연계: 이미 연동된 상품 중 성과 하락(품절 전환·중지)도 부활 후보로 재승격.

## 작업 유의사항/원칙 (신규)
- **#244** 상품 연동/import는 "전체 복제"가 아니라 "개선 필요 상품 선별→앱 튜닝→재업로드" 파이프라인. 앱은 **부활 대상 자동 식별(가용 신호 스코어링)**로 운영자의 선택 부하를 줄여야 함(#56). 외부 미제공 데이터(리뷰수 등)는 스코어에서 정직히 제외(우회·크롤링 금지·#231). 선별 import→부활소→아틀리에→발행 준비→재업로드가 하나의 개입점 흐름으로 연결(product-agnostic).
