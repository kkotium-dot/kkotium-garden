## 2. 개입점 4유형 (어디서 대표가 필요한가)
- 결정(Decision): 큐레이션 · 모드/라인 · 대표 이미지 승인 · SEO 검수
- 입력(Input): 인증번호 · 미보유값 · 소스 이미지 업로드
- 인증(Auth): Firefly/네이버 로그인 · CAPTCHA
- 비가역 GO: 네이버 PUT/POST (#46 — 명시 "GO" 전 금지)

## 3. 오케스트레이션 — 개입 대기열 (Operator Action Queue)
시스템의 핵심. "저의 개입이 자연스럽게"의 실체.
- 전상품을 가로지르는 단일 surface. "지금 무엇이 · 어느 상품에 · 왜 필요한가"를 카드로 노출.
- 상태 4분류: 자동 진행(초록) / 입력·결정 대기(앰버) / 비가역 GO 대기(레드) / 외부 인증 필요(일시정지).
- 각 카드 = 상품명 + 필요 행동 1줄 + 1클릭 해당 화면 진입. 순서 강제 0(상황 따라 융통적 처리).
- 적용 현황(#54)과 한 쌍: 적용현황="무엇이 됐나"(결과축) · 개입 대기열="무엇을 해야 하나"(행동축).
- 데이터 원천: control-tower-engine의 nextAction + applyStatus + publishState 파생(신규 컬럼 0, 가드 불요).

## 4. 원칙 #56 — 개입 자연스러움 (smooth human-in-the-loop)
- 대표 개입이 필요한 모든 지점은 앱이 자연스럽게 surface한다(숨김·추측 강요 금지).
- 기능은 상황에 따라 융통적으로 사용(순서 강제 X). #54(적용현황 상시)·#55(전상품 범용)와 통합 운용.
- 자동/반자동 경계: 비가역·인증·창작 판단은 대표; 그 외 의존성 없는 기술작업은 앱/Code가 완결 후 보고.

## 5. 구현 로드맵 (전상품, 점진 — 과잉 한 번에 X)
- 1차: control-tower-engine에 actionQueue 파생(nextAction + applyStatus + 게이트 → 4분류) + 관제탑 상단 "개입 대기열" 위젯(1클릭 진입). 기존 신호 재사용, 신규 컬럼 0.
- 2차: 각 개입점 화면 진입 핸드오프(딥링크) 정합 — 큐 카드 클릭 → 정확한 작업 화면/탭.
- 3차: 외부 인증 개입(Firefly/네이버)을 Chrome MCP 반자동(#52 detect→deliver→resume)과 연결 — CAPTCHA·로그인·다운로드만 대표, 나머지 Claude.
