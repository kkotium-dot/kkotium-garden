### 두 화면이 동일

| 요소 | 정원 창고 (?tab=draft) | 꽃밭 돌보기 (/products) |
|---|---|---|
| 헤더 | 목록·공급사별·**네이버 동기화**·**네이버 상품 가져오기**·상품 등록 | 동일 |
| 필터 | **판매중 0 · 판매중지 1 · 좀비꽃 발견 1** · 작성중 2 | 동일 |
| 컬럼 | 상태·공급사·배송·순마진·판매가·준비도·점수·관리 | 동일 |
| 행 배지 | 앱생성·미등록·발행 8/8·상품명 S·**잘 자라는 중**·**재고 확인 실패** | 동일 |

### 정원 창고에서 말이 안 되는 것 (실측 확인)

| 항목 | 왜 틀렸나 |
|---|---|
| **네이버 동기화** 버튼 | 정원 창고 상품은 네이버에 없음(naverProductId 없음) — 동기화 대상 0 |
| **네이버 상품 가져오기** 버튼 | 네이버에서 가져오면 naverProductId가 생겨 **꽃밭 돌보기 소관**이 됨. 정원 창고에 둘 이유 없음 |
| **판매중 0 / 판매중지 1 / 좀비꽃 발견 1** | ★**다른 화면(꽃밭 돌보기)의 건수를 보여줌**. 정원 창고에는 해당 상품이 하나도 없는데 필터가 노출됨 |
| **"잘 자라는 중"** 배지 | 좀비 판정은 판매 실적·마진·트렌드 기반(#264). 안 판 상품엔 무의미 |
| **"재고 확인 실패"** | 네이버 재고 폴링 결과(#260). 네이버에 없는 상품을 물어본 것 |
| **순마진 15.3%** | 실적이 아니라 예상치인데 꽃밭 돌보기와 같은 라벨로 표기 → 오해 유발 |

### 꽃밭 돌보기에서 불필요한 것

| 항목 | 왜 |
|---|---|
| **상품 등록** 버튼 | 신규 등록은 온실아틀리에 소관. 꽃밭은 이미 발행된 상품을 돌보는 곳 |
| **준비도** 컬럼 | 이미 발행됐으므로 발행 준비도는 의미 없음 |
| **작성중 N** 필터 | #266 교정으로 정원 창고 소관이 됨 |

## 4. 설계안

### 4-1. 헤더 액션

**정원 창고**
```
[+ 상품 등록]  [크롤링에서 가져오기]  [준비된 것 일괄 발행 N]
```
- `네이버 동기화` **제거** (대상 0)
- `네이버 상품 가져오기` **제거** (가져오면 꽃밭 소관)
- `준비된 것 일괄 발행` **신설** — 준비도 100% 상품을 한 번에. 정원 창고의 존재 이유
- 버튼에 건수 표시 → "오늘 3개 올릴 수 있어요"가 한눈에

**꽃밭 돌보기**
```
[네이버 상품 가져오기]  [네이버 동기화]
```
- `상품 등록` **제거** (온실아틀리에 소관)
- 기존 유지

### 4-2. 필터

**정원 창고** — 작업 단계 기준 (판매 상태 아님)
```
● 준비 미흡 N   ● 발행 가능 N   [전체 N ▾]
```
- `준비 미흡` = 준비도 < 100%
- `발행 가능` = 준비도 100% → **바로 올릴 수 있는 것**
- 판매중/판매중지/좀비꽃 **제거** (해당 상품 0건)

**꽃밭 돌보기** — 판매 상태 기준 (현행 유지)
```
● 판매중 N   ● 판매중지 N   ● 좀비꽃 발견 N   [발행 전체 N ▾]
```

### 4-3. 컬럼

| 정원 창고 | 꽃밭 돌보기 |
|---|---|
| 상품명 / SKU | 상품명 / SKU |
| **작업 단계**(씨앗·꽃단장·발행대기) | **판매 상태**(판매중·판매중지·품절) |
| 공급사 | 공급사 |
| — | 배송 |
| **예상 마진**(라벨에 "예상" 명시) | **순마진**(실적) |
| 예상 판매가 | 판매가 |
| **준비도** ★핵심 | — |
| 상품명 점수 | **좀비 지수** |
| 관리 | 관리 |

### 4-4. 행 배지

| 정원 창고 | 꽃밭 돌보기 |
|---|---|
| ✅ 발행 8/8 (준비 항목) | ✅ 좀비 발견 / 잘 자라는 중 |
| ✅ 상품명 S·우수 | ✅ 재고 N개 |
| ✅ 앱생성 / 크롤링 | ✅ 연동 / 앱등록 |
| ❌ **좀비 배지 제거** | ✅ 판매중지 |
| ❌ **재고 확인 실패 제거** | ✅ 상품명 등급 |

### 4-5. ★시스템 레벨 교정 (단건 아닌 전체 적용·#62)

**좀비 판정 대상을 스토어 등록 상품으로 제한**

현재 `loadTuningScores`가 **미발행 상품에도 좀비 점수를 계산**하고 있음.
- 좀비 = "되살릴 수 있는 상품"(#264) = **팔았던 적 있는 상품**
- 안 판 상품은 되살릴 것도 없음 → 판정 자체가 무의미
- 낭비: 미발행 상품마다 마진·트렌드·품절대체 조회가 헛돎

**조치**: `zombie-verdict.ts`에 `!naverProductId면 판정 스킵` 가드 추가.
영향 범위(전수 확인 필요):
- `/products` 행 배지
- 대시보드 좀비 카운트 (`dashboard/stats`)
- 사이드바 배지
- 디스코드 좀비 알림
→ **한 곳에서 고치면 4곳이 같이 정상화됨**(#62)

**네이버 재고 폴링도 동일** — `!naverProductId`면 폴링 스킵(#260 연장).

### 현재 상태 (prod ee77032, src/app/products/page.tsx 실측)

- **라벨은 이미 분기됨** (1929~1930행):
  - 정원: `상품명 / 작업 단계 / 공급사 / 배송 / 예상 마진 / 예상 판매가 / 준비도 / 점수 / 관리`
  - 꽃밭: `상품명 / 상태 / 공급사 / 배송 / 순마진 / 판매가 / 준비도 / 점수 / 관리`
- **그러나 그리드 자체는 공유** (1792행):
  `const COL = '36px 1fr 90px 110px 130px 62px 90px 72px 68px 70px'`
  → 두 화면이 같은 컬럼 수/폭을 씀. 라벨만 바뀌고 실제 열 구성은 동일.

### 문제

1. 정원 창고에 **배송 컬럼**이 있으나 미발행 상품엔 배송이 의미 없음(아직 안 팔림)
2. 꽃밭 돌보기에 **준비도 컬럼**이 있으나 이미 발행된 상품엔 준비도 무의미
3. 꽃밭 돌보기 **점수**가 상품명 점수인데, 발행 후엔 좀비 지수가 더 중요

### 구현 주의 (왜 별도 턴이 필요했나)

- `COL` 상수가 헤더(1929)·행(1801)·일괄발행 헤더(1920) **3곳에서 공유**됨.
  뷰별로 컬럼 수가 달라지면 `COL`을 `isGarden` 분기해야 하고, 셀 렌더 순서도
  그에 맞춰 조건부로 재배치해야 함. 그리드-셀 정합이 깨지면 열이 밀림.
- **접근법 제안**: `COL_GARDEN` / `COL_CARE` 두 상수로 분리 →
  헤더/행/일괄발행헤더 3곳 모두 `isGarden ? COL_GARDEN : COL_CARE` →
  각 행의 셀 렌더도 배송/준비도 셀을 `isGarden` 조건부로 감쌈.
- 회귀 위험 높음: 반드시 tsc0/build0 후 **정원·꽃밭 양쪽** DOM 실측으로
  열 개수·라벨·정렬 확인. 운영자 육안 필수(#265).
