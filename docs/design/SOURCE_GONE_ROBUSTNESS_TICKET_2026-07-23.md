# sourceGone 강건성 티켓 — 설계안 (구현 전, 감사 결과)

> 상태: 설계만. 코드 미변경. 작업원칙 #62(단일 권위) 유지 전제 — 고칠 지점은
> `src/lib/products/source-gone.ts` 단 하나(아래 §4 blast radius 참조).

## 1. 문제

`countLeadingNegatives()`(source-gone.ts:29-40)는 최신순(desc) 스냅샷을 훑다가
**첫 양수 qty를 만나는 순간 해당 상품의 런(run)을 완전히 봉인(seal)**한다.

```ts
if (s.qty < 0) counts.set(s.productId, (counts.get(s.productId) ?? 0) + 1);
else sealed.add(s.productId); // first non-negative seals the run
```

`isSourceGoneFromCount()`는 이 카운트가 `SOURCE_GONE_MIN_CONSECUTIVE`(=3) 이상일
때만 `sourceGone=true`를 반환한다. 즉 판정은 **"최신 스냅샷부터 연속으로 몇 번
음수였는가"** 단 하나의 신호에 전부 의존한다.

## 2. 구체적 실패 시나리오

폴러가 6시간 주기로 도는 상황을 가정(작업원칙 #62 문서 상단 배경: 소싱 단절은
연속 3회 qty<0으로 확정).

| 폴 순서(최신→과거) | qty | 설명 |
|---|---|---|
| 1 (최신) | -1 | 조회 실패 |
| 2 | -1 | 조회 실패 |
| 3 | **1** | 폴러가 캐시/타임아웃 레이스로 순간적으로 양수 1건 오독 |
| 4 | -1 | 조회 실패 |
| 5 | -1 | 조회 실패 |
| 6 | -1 | 조회 실패 (공급사가 실제로 상품을 내린 지 5폴 경과) |

실제로는 상품이 5번 연속(폴 2~6 구간에 낀 1건 제외) 조회 실패 상태지만, 현재
로직은 최신 스냅샷(폴 1)부터 세다가 폴 3의 단발성 양수 1건에서 즉시 seal한다.
→ `count = 2` → `2 < 3` → **`sourceGone = false` (오판)**.

이 오판은 `decideDisposition()`의 최우선 분기(disposition.ts:121, "소싱 단절은
확정 신호… 최우선(#271 사상)")를 통째로 건너뛰게 만든다. 결과적으로:

- `hasAssets=true`인 상품이면 **RESOURCE(대체 소싱) 권고가 뜨지 않고 조용히
  방치**된다 — 정확히 disposition.ts 주석이 경계하는 "자산 보호(#272) 실패"
  상황.
- 폴러가 "실단절 중 일시 양수 1건"을 다시 읽지 않는 한(=공급사가 실제로 상품을
  되살리지 않는 한) 카운트는 영원히 3에 도달하지 못하고 `false`로 고착된다 —
  단순 지연이 아니라 **영구 오판**.

`qty<0`이 실제로 무엇을 의미하는지(조회 실패 센티널, #260)를 감안하면, 크롤러
파싱 흔들림·프록시 타임아웃·도매꾹 쪽 순간 응답 지연 등으로 단발성 "가짜 양수"가
섞여 들어올 개연성은 낮지 않다.

## 3. 옵션 비교

### 옵션 A — 현행: "연속 N" (strict consecutive-from-newest)

- 로직: 최신부터 훑다가 첫 비음수에서 seal.
- 장점: 구현 단순, 설명 쉬움("최근 N번 연속 실패").
- 단점: §2처럼 단일 노이즈 샘플에 취약 — **1건의 오독이 판정을 무한정 되돌림**.

### 옵션 B — "최근 M 중 N 음수" (windowed threshold)

- 로직: 최신 M개 스냅샷을 보고, 그중 음수가 N개 이상이면 `sourceGone=true`.
  예: M=5, N=4.
- 장점: 단발성 노이즈에 강함. 통계적으로 "대체로 죽어있다"를 포착.
- 단점:
  - **"연속"이라는 확정 신호의 성격이 흐려진다** — disposition.ts는 소싱단절을
    "다른 어떤 조건으로도 상쇄되지 않는 확정 신호"로 취급하는데(§1 사상,
    #271), M-of-N은 본질적으로 통계적 근사이며 threshold 튜닝이 필요해진다.
  - 상품이 실제로 막 재입고된 경우(최신 2~3폴 연속 양수)에도 과거 창의
    음수가 많으면 오탐(false positive)으로 sourceGone이 계속 유지될 수 있음
    — 최신 상태 반영이 늦어짐(latency 방향이 반대로 뒤집힘).
  - M, N 두 파라미터를 동시에 튜닝해야 함 (현재 3 하나보다 설계 표면 증가).

### 옵션 C — "연속, 단 단발성 이상치 1건 허용" (tolerant-consecutive) — 권장

최신부터 훑는 "연속" 정신은 유지하되, **런 내부에서 고립된 단일 양수(앞뒤가
모두 음수인 1개짜리 스파이크)는 seal 신호로 취급하지 않는다.**

- 규칙: 최신부터 순회. 양수를 만나면 즉시 seal하지 않고 "보류(pending)"
  상태로 표시한 뒤 **다음 스냅샷까지 확인**한다.
  - 다음 스냅샷도 양수 → 진짜 회복(재입고) → seal 확정, 카운트 리셋.
  - 다음 스냅샷이 음수 → 방금 양수는 고립 스파이크로 간주, 카운트에 넣지
    않고(=벌점 없음, 가산도 안 함) 계속 진행.
  - 창(window) 안에서 이 "고립 스파이크 무시"는 **런당 최대 1회**로 제한한다
    (2건 이상 연속 양수는 무조건 회복으로 간주 — 옵션 A의 확정성을 대부분
    보존).
- 장점:
  - "연속 = 확정 신호" 원칙(#271)을 사실상 유지 — 진짜 회복(2폴 이상 연속
    양수)은 여전히 즉시 반영됨.
  - 노이즈 내성 추가 폭이 작고 검증 쉬움("한 번의 튐은 무시, 두 번째부터는
    진짜 재입고로 인정").
  - 파라미터 추가 없음(기존 `SOURCE_GONE_MIN_CONSECUTIVE=3` 그대로 재사용
    가능) — 튜닝 표면 최소화.
- 단점: 판정에 1폴(6시간) 정도의 확인 지연이 항상 붙는다(양수 1건만으로는
  즉시 회복 처리하지 않으므로). 다만 이건 "재고 있음 반영이 6시간 늦어지는
  것"이지 "품절인데 방치되는 것"보다 훨씬 안전한 방향의 지연이다.

## 4. 권장안 및 blast radius

**옵션 C 권장.** 이유: disposition.ts가 sourceGone을 "다른 조건으로 상쇄되지
않는 확정 신호"로 못박아 놓았기 때문에(§1 설계 판단 1번), 옵션 B처럼 확률적
threshold를 들여오면 그 확정성 전제가 깨진다. 옵션 C는 확정성을 유지하면서
가장 흔한 실패 모드(단발성 오독 1건)만 방어한다.

수정 지점은 `countLeadingNegatives()` 단 하나(source-gone.ts:29). 소비처
3곳(`inventory-badges` route, `tuning-signals.ts` 좀비 판정,
`disposition-load.ts` → KkottiWidget/notifications/weekly-report/cron-daily
전체)이 모두 이 함수를 거치므로, 여기 한 곳만 고치면 전 채널에 동시 반영된다
— 단, **역으로 이 함수 하나의 버그가 전 채널에 동시에 잘못 퍼진다는 뜻이기도
하므로 테스트가 특히 중요하다(§5).**

## 5. 테스트 계획

리포에 테스트 프레임워크가 없다(package.json에 `test` 스크립트 없음). 기존
관례(`src/lib/naver/category-score.test.ts` 등, `node:assert/strict` +
`npx tsx` 단독 실행) 그대로 따라 `src/lib/products/source-gone.test.ts` 신설.

케이스:

1. **기존 동작 보존** — 순수 연속 음수 3회 → `sourceGone=true` (회귀 방지).
2. **§2 시나리오** — `[-1,-1,+1,-1,-1,-1]`(최신→과거) → 옵션 A라면 count=2/false,
   옵션 C 적용 후 기대값 = 고립 스파이크 무시 → 유효 연속 5 → `true`.
3. **진짜 회복 인식** — `[-1,-1,+1,+1,-1,-1,-1]` → 최신 2건이 연속 양수이므로
   즉시 seal, `sourceGone=false` (재입고를 놓치면 안 됨).
4. **런당 스파이크 1회 제한** — `[-1,+1,-1,+1,-1,-1,-1]` (스파이크 2개, 서로
   떨어져 있음) → 스펙 확정 필요: 이미 1회 소진 후 두 번째 스파이크는 정상
   seal 처리되어야 하는지 여부를 운영자와 §3 확정 시 함께 결정.
5. **데이터 없음 / 신상품** — 스냅샷 0건 → `sourceGone=false`(기존과 동일,
   "모른다고 처분 권고 안 함" 원칙 disposition.ts:131-133과 정합).
6. **`loadSourceGoneFlags` best-effort 유지** — DB 에러 시 여전히 조용히
   degrade(source-gone.ts:97-99, #82)하는지 회귀 확인.

## 6. 확정 필요 사항 (운영자 결정 대기)

- §5 케이스 4의 "런당 스파이크 1회" 경계 조건(스파이크가 2개 이상 떨어져
  있을 때 각각 허용할지, 런 전체에 1회만 허용할지) — 어느 쪽이든 구현
  가능하나 정책 성격의 판단이라 확정 후 착수.
- 창(window) 지연 수용 여부: 재입고 반영이 항상 최소 2폴(≈12시간) 걸리는
  트레이드오프에 대한 최종 승인.

이 문서 확정 후 구현은 별도 세션/승인으로 진행.
