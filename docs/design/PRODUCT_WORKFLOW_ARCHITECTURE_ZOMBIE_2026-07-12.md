# 상품 통합 워크플로우 아키텍처 — 씨앗심기 중심 + 좀비 부활 엔진 (2026-07-12)

Authoring: DESKTOP(프리미엄 B2B SaaS 설계). 계기: 운영자 IA 전면개편 보충. 원칙 #231/#233/#245/#249/#251/#254. 상태: ★설계 브리핑·승인 후 단계 구현.

## 0. 확인된 제약 (정직·#231)
- 네이버 Commerce API = 판매자 상품 데이터(statusType·salePrice·stockQuantity·category)만. **공식 품질지수·검색순위·유입·리뷰수 미제공.**
- 우리가 가진 데이터: 주문/판매실적(Order sync)·재고/가격/상태(Naver)·상품명 SEO(자체 진단)·카테고리 트렌드(DataLab)·공급가/경쟁가(dome analyzer).
- → 네이버 점수 못 가져오므로 **자체 '튜닝 필요도' 지수**로 우회(정직 표기).

## 1. 시스템 아키텍처 — 톱니바퀴(맞물림)
```
      [발굴]                [제작·수정 엔진]            [통합 관리·감시]
   꿀통 꽃나들이   ──신규──▶   씨앗심기(중앙 기어)  ──발행──▶  등록 상품 목록
   (소싱 후보)                  ▲        │                    (연동+앱생성 통합)
                               │        │                        │
                          수정 │        │ 임시저장/등록           │ 감시
                               │        ▼                        ▼
                         [수정 버튼]◀──── 좀비 감지 엔진 ◀───── 튜닝 필요도 지수
   네이버 스토어 ──연동 import──▶ 등록 상품 목록          (품절·악성재고·마진위기·SEO미달)
                                       │                         │
                                       └──────좀비 배지+디스코드 알림◀┘
```
- **씨앗심기 = 중앙 기어**: 신규 등록·연동 수정·좀비 부활 — 모든 편집이 여기로 수렴(단일 편집 엔진).
- **등록 상품 목록 = 통합 관리+감시**: 연동+앱생성 구분 없이 한 목록. 좀비 감지 엔진이 상시 감시.
- **좀비 감지 엔진**: 튜닝 필요도 지수 계산→임계 초과 시 좀비 배지+디스코드 알림→[수정 바로가기]→씨앗심기.
- **순환**: 발행된 상품이 좀비화되면 다시 씨앗심기로(리셋 튜닝)→재발행. 정원 은유: 발굴→가꾸기→(시들면)부활.

## 2. 씨앗심기 중심 수정 흐름도 (플로우차트)
```
등록 상품 목록 (판매중/판매중지/좀비발견 필터)
   │  상품 클릭
   ▼
상품 정보 화면 (연동상품=네이버 원본 최대 표시·좀비면 "왜 좀비인지" 한 줄 이유)
   │  [수정] 버튼
   ▼
씨앗심기 진입 (기존 정보 전부 품고 이동) ── 3-레이어 편집 ──┐
   ├─ [네이버 원본 정보]   읽기전용 (연동해온 원본)          │
   ├─ [업데이트할 정보]    앱 개선분 (상품명·이미지·가격 등)   │
   └─ [품절 시 대체 상품]  대체 상품 지정                     │
   │                                                        │
   ▼  선택                                                   │
 [임시저장] ─▶ 목록에 '작성중'으로 보존 (나중에 이어서)         │
 [등록/재발행] ─▶ 발행 준비 체크(8/8) ─통과─▶ 네이버 반영(GET-merge+GO)
                        │미달                                 │
                        └─▶ 부족 항목 표시(#240) ◀────────────┘
```

## 3. 튜닝 필요도 지수 (자체 지수·셀러 언어)
네이버 점수 대신, 가용 신호 가중합 → 0~100 (높을수록 손봐야 함). UI 라벨은 외계어 배제:
| 신호(내부) | 셀러 표현 | 방향 | 데이터 |
|---|---|---|---|
| 판매속도 하락/0 | "요즘 안 팔림" | ↑ | Order 실적(7/30일) |
| 품절·재고0 | "품절" | ↑↑ | Naver stock |
| 장기 재고 미변동 | "창고에 잠든 상품" | ↑ | 재고 스냅샷 |
| 마진 위기(공급가↑/경쟁가↓) | "남는 게 줄었어요" | ↑↑ | dome analyzer |
| 상품명 SEO 낮음 | "검색 노출 약함" | ↑ | diagnoseProductName |
| 카테고리 트렌드 하락 | "시즌 지남" | ↑ | DataLab |
| 오래 방치·미수정 | "손본 지 오래" | ↑ | updatedAt |
| (리뷰수) | — | 제외 | API 없음(#231) |
- 임계 초과 → **좀비 배지** + 디스코드 알림. caveat 정직 표기("네이버 공식점수 아님·자체 추정").

## 4. 좀비 부활 우선순위 (매출영향 × 시급성)
1. **P1 품절+판매실적 상품** — 기회손실 최대. 즉시 대체상품 or 재입고.
2. **P2 마진 위기** — 경쟁가 하락/공급가 상승. 손실 방어(가격·소싱 재검토).
3. **P3 성장 여력**("잘 팔리는데 SEO/트렌드로 더 키울") — 상승 상품 부스트.
4. **P4 장기 악성 재고**(판매0·방치) — 정리 or 부활 결정.
→ 목록/알림에서 이 순서로 정렬. 꼬띠: "지금 부활시키면 마진 00% 확보!"

## 5. 품절 시 대체 상품 (즉각 조치 UI)
- 재고 0 순간 → **빨간 강조 "대체 상품 지정" 폼** 팝업(#56 개입점).
- 입력 방식(제안): ⓐ앱 등록 상품 목록에서 선택(연결) ⓑ도매매 상품코드 입력 ⓒ같은 카테고리 자동 추천 중 선택.
- 대체 지정 시: 상세페이지에 "품절 — 이 상품 추천" 노출 or 옵션 자동 전환. 기회손실 최소화.

## 6. 통합 등록 상품 목록 (수백 개 대비)
- 연동+앱생성 구분 없이 한 목록(#245). 카드 그리드·정렬 통일·핵심정보(썸네일·상품명·상태·좀비 배지+한줄이유·다음 액션 1개).
- 상태 필터 심플: **판매중 / 판매중지 / 좀비발견** (+필요시 품절·작성중). 나머지 드롭다운.
- 수백 개 대비: 서버 페이지네이션(`< 1 2 3 4 … >`)·가상 스크롤·검색·정렬(튜닝 필요도순).
- 연동 선택 화면도 동일: 판매중/판매중지 + **튜닝 필요도 구분**(키울 상품/방어/내릴 상품).

## 7. 나의 추가 아이디어
- **"다음 할 일" 1버튼**(카드마다): 앱이 판단(#240). "뭐 하지?" 제거.
- **포커스 모드**: 상품 클릭→그 상품 전용 작업공간(편집·SEO·발행 한 맥락). 메뉴 오가기 0.
- **좀비 알림=행동형**: "좀비 출현! 지금 부활 시 마진 00%·[수정 바로가기]"(딥링크·꼬띠 보이스).
- **자동 재발행 준비**: 씨앗심기 수정→발행 준비 8/8 자동 체크→GO만 누르면 재업로드.

## 작업 유의사항/원칙 (신규)
- **#255** 네이버 미제공 지표(품질지수·유입·리뷰)는 가져올 수 있는지 먼저 API 확인 후, 불가하면 자체 가용 데이터로 독자 지수(튜닝 필요도) 구성하되 "공식 점수 아님" 정직 표기(#231). 모든 편집은 씨앗심기 단일 엔진으로 수렴(중앙 기어)·목록은 연동/앱생성 통합·상태 필터 최소·좀비는 배지+한줄이유+행동 알림. 지수/우선순위는 매출영향×시급성. UI는 외계어 배제·셀러 언어(#233).
