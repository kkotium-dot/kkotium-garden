## 0. 목적 (왜)
================================================================
스튜디오(S2 이미지 생성/가공)와 발행(S5) 사이의 **수작업 Figma/Canva 조립**을 앱 내부로 흡수.
상품별 가공 자산(stage)을 **7섹션 860px 상세페이지**로 조립하는 단일 보드. 표준 케이스를 1인 셀러가 드래그만으로 양산 → 파워셀러 속도.

- 파이프라인 위치(OPERATOR_SYSTEM_BLUEPRINT §1): 4(이미지)→5/6(SEO/속성)→**조립**→7(검수)→8(발행).
- 출력 배관은 이미 존재(IMAGE-SPLIT): detail_images(jsonb) + buildDetailContent HTML. 본 보드는 그 입력을 **사람이 조립**하는 UI.
- 데이터 입력원: GET /api/products/{id}/assets — **8스테이지**(source/cutout/plate/**reference**/composite/thumbnail/detail/archive) 구조 + publicUrl (Desktop 실측 검증완료, storage 직독). ★스테이지는 asset-taxonomy.ts STAGE_DIRS가 코드진실 = 8종.

## 작업 유의사항/원칙
================================================================
- **#182** AtelierShell은 활성 사이드바 탭 content만 마운트(`{activeTab.content}`) — 특정 탭이 안 보인다고 미렌더 버그로 단정 금지. 탭 전환 후 검증(SF-1 1차 진단 혼선의 교훈: 도구함 탭에서 배양실 콘텐츠를 찾던 착오).
- **#181** 신규 기능은 기존 출력 배관(detail_images/detailContent)·데이터소스(assets API) 위에 설계 — 중복 구축 금지. 보드는 새 저장/발행이 아니라 사람-조립 입력 UI.
- **#70** 슬롯/스테이지 택소노미는 asset-taxonomy.ts가 코드진실(8종) — 스펙과 불일치 시 코드 우선(v1.1에서 정합).
- 신규 보드는 ErrorBoundary로 격리(#62) — 렌더 오류가 전체 캔버스를 침묵으로 날리지 못하게.
