## 0. 핵심 철학

> 클로드 디자인으로 모든 걸 끝내는 게 아니다. **언제·어떻게 활용하느냐**가 핵심.
> 클로드 디자인 = 매일 돌리는 공장이 아니라 *고급 마감 작업대*.
> "자동화가 감각을 대체하는 게 아니라, 감각을 펼칠 시간을 벌어준다."

대표님이 짚은 결정적 사실: **네이버 상세페이지는 HTML 직접 등록 가능** → 클로드 디자인 HTML 출력물을 그대로 활용하는 길이 열림.

## 1. 확정 워크플로우 (대표님 확정 흐름)

```
[1단계] 소재 준비
  어도비 Firefly  → 완성도 높은 합성 AI 이미지 (실사 상품 + 감성 배경)
  아티팩트(Claude) → 상세페이지 구조 설계 + 텍스트 흐름
        │
        ▼
[2단계] 클로드 디자인 상세페이지 작업
  1단계의 이미지 + 설계된 구조를 입력
  꽃틔움 디자인 프리셋(디자인 시스템) 자동 적용
  슬라이더 / 직접 편집 / 코멘트로 디렉팅
        │
        ▼
[3단계] 상황별 마무리 (두 갈래)
  길 A (단건·특별): 필요한 확장자로 마무리 → 네이버 직접 업로드
  길 B (대량·일상): 추가 디자인 작업 또는 자체 엔진 컴포넌트화
```

---

## 2. 도구별 역할 분담 (조사 근거 반영)

| 단계 | 도구 | 역할 | 근거 (조사 결과) |
|---|---|---|---|
| 소재-이미지 | 어도비 Firefly | 실사 상품 합성, 감성 배경 | 클로드 디자인은 *이미지 생성 불가* → Firefly가 담당 |
| 소재-구조 | 아티팩트 | 상세페이지 구조·텍스트 흐름 설계 | 빠른 시안·구조 탐색에 적합 |
| 설계-프리셋 | 클로드 디자인 (1회성) | 꽃틔움 디자인 시스템 등록·학습 | "Teach Claude your brand" 기능 확인 |
| 설계-시안 | 클로드 디자인 (탐색) | 이미지+구조로 상세페이지 시안 | 출력물이 *HTML 코드* (픽셀 아님) |
| 마무리 | 클로드 코드 | HTML 완성 → 네이버 등록 or 컴포넌트화 | Handoff 번들 = 토큰+구조+의도 포함 |
| 대량생산 | 자체 엔진 (Satori+Sharp) | props 갈아끼워 매일 N개 자동 | 클로드 디자인 API 없음 → 일상 운영 부적합 |

## 3. 두 갈래 마무리 전략 (대표님 핵심 통찰)

대표님 발화: "상황에 따라 마무리를 하거나, 제가 직접 작업하는 상황이 오더라도..."

### 길 A — 빠른 단건 (소량·특별 상품)
```
클로드 디자인 HTML 받기 → 클로드 코드로 다듬기 → 네이버 HTML 직접 등록
```
- 용도: 새 카테고리, 시즌 신상, A급 단건, 특별 기획
- 장점: 정성·완성도. 클로드 디자인의 디렉팅 강점 100% 활용
- 비유: 특별 손님상은 셰프가 직접 플레이팅

### 길 B — 대량 반복 (일상 상품)
```
클로드 디자인 시안 → 클로드 코드로 컴포넌트화 → 자체 엔진 매일 자동 생산
```
- 용도: 검증된 일상 상품, 대량 등록
- 장점: 비용 0, 매일 N개, 클로드 디자인 한도 소모 0
- 비유: 매일 메뉴는 레시피(컴포넌트)로 주방이 자동 조리

**대표님이 상황에 따라 길 A/B를 선택.** 둘은 배타적이지 않고 상호 보완.

## 4. 프리셋 동기화 고리 (시니어 제안 — 일관성 보장)

클로드 디자인에서 확정한 *꽃틔움 디자인 프리셋(색·폰트·여백 토큰)*을,
클로드 코드를 통해 **자체 엔진의 디자인 토큰과 동일하게** 맞춤.

```
클로드 디자인 프리셋 (본점 인테리어 색)
        │  클로드 코드가 토큰 추출·동기화
        ▼
자체 엔진 디자인 토큰 (분점이 같은 페인트 코드로 칠함)
```

효과: 길 A로 직접 만든 상세페이지와, 길 B로 자동 생산된 상세페이지가
*형제처럼 일관된 꽃틔움다움* 유지.

## 5. 클로드 디자인 활용 원칙 (조사 기반 한계 반영)

| 원칙 | 이유 (조사 확인) |
|---|---|
| 매일 호출하지 않는다 | 주간 한도 빠른 소진 (Max5도 power 사용 수준) |
| 이미지 생성은 기대 안 한다 | 이미지 생성 기능 없음 → Firefly가 담당 |
| 설계·마무리·디렉팅에만 쓴다 | API 없음, 웹 전용. 자동화 부적합 |
| 한국어 폰트는 명시 지정 | Pretendard는 폰트 파일/CDN URL 명시 필요 |
| 출력은 HTML로 받아 활용 | 픽셀 아닌 코드 출력 → 네이버 HTML/자체 엔진 양쪽 호환 |
| 프리셋은 깨끗하게 1개로 등록 | 큰 코드베이스 업로드 시 한도 소진·렉 |
