# COLLABORATION_PLAYBOOK — 문제 해결 방식 지침

> 운영자가 문제 해결을 요청할 때, **가장 저렴하고 되돌리기 쉬운 방안부터** 제안하기 위한 판단 기준.
> 2026-07-23 신설. 실사고에서 도출된 규칙만 담는다.

## 1. 해결안 제안의 우선순위 (항상 이 순서로 검토)

| 순위 | 방식 | 이유 |
|---|---|---|
| 1 | **기존 것을 고쳐 쓴다** | 누적 이력·메모리·설정이 보존된다 |
| 2 | **기존 것에 덧붙인다** | 되돌리기 쉽다 |
| 3 | 새로 만든다 | 최후. 이력이 끊기고 관리 지점이 늘어난다 |

**실사례**: Cowork가 코드를 못 보는 문제 → "새 프로젝트 생성"(3순위)이 아니라 **"기존 프로젝트에 폴더 추가"**(2순위)가 정답. 새로 만들면 프로젝트 단위로 격리된 메모리와 작업 이력이 통째로 끊긴다.

## 2. 설정은 계층을 분리한다 (섞으면 둘 다 무뎌진다)

| 계층 | 담을 것 | 담지 말 것 |
|---|---|---|
| **전역(상시 지침)** | 태도·정직 규칙·보고 형식 | 프로젝트 이름·저장소 사실 |
| **프로젝트(폴더 지침)** | 도메인 사실·코드 규칙·현재 스프린트 | "친절하게 답하라" 같은 태도 |

- 전역에 프로젝트 사실을 넣으면 → 다른 일 할 때 잘못된 전제가 따라온다.
- 프로젝트에 태도 규칙을 넣으면 → 프로젝트마다 복붙, 하나 고치면 전부 어긋난다.
- 프로젝트 지침에 **진행 상태를 적지 말고 "가리키게"** 한다("최신 rev = 현재 상태"). 트래커만 갱신하면 지침이 자동으로 최신이 된다.

## 3. 프로세스보다 환경으로 먼저 막는다

인계 문구로 당부하는 것보다 **접근 자체를 열어주는 것**이 확실하다(#304). 같은 사고가 재발하면 "더 강하게 당부"하지 말고 **설정을 바꾼다.**

## 4. 커넥터가 막히면 대체 경로를 전수 시도한다 (#298)

Filesystem MCP ↔ Desktop Commander ↔ iterm ↔ Chrome 커넥터는 **상호 백업**이다. 하나가 죽어도 다른 것으로 읽기·쓰기·검증이 가능하다.
- 4분 행(#26) 패턴은 **재시도하지 말고 즉시 다른 커넥터로 전환**한다.
- 전수 시도 전에 "못 한다"고 보고하지 않는다. 단, 전부 실패했으면 **지어내지 말고 사실대로 보고**한다.

## 5. 검증은 "읽었냐"가 아니라 "읽어야만 답할 수 있는 것"으로 한다

연결·적용 여부를 확인할 때는 파일을 실제로 읽지 않으면 절대 못 맞히는 값을 묻는다(최신 rev 번호·마지막 원칙 번호·특정 컬럼 존재 여부·정확한 파일명). "읽었습니다"라는 답변은 증거가 아니다.

## 6. 문서와 구현이 어긋나면 구현을 의심하기 전에 문서를 의심한다

2026-07-23 사례: 브리지 문서가 "상태 9종"이라 했으나 표에는 7종뿐이었고, 구현(7종)이 정답이었다. 지시받은 쪽이 **지어내지 않고 멈춰 확인 요청한 것이 옳았다**(#303). 숫자·목록이 본문과 어긋나면 즉시 멈추고 확인한다.
